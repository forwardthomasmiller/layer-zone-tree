# 后记：一个 Vibe Coder 的架构感悟

写完这本书，我想再说几句掏心窝子的话。

## 我不是架构专家

这一点我想说清楚。

我不是计算机科班出身，没有系统学过软件工程，在做 GetDone Timer 之前甚至没写过 macOS 代码。分层架构、设计模式这些概念，我都是在做项目的过程中边踩坑边学的，谈不上系统掌握。

我是一个 Vibe Coder。一个靠 AI 写代码、靠直觉做产品的人。

Layer-Zone Tree 不是什么精心设计的架构理论。它是我在实战中被逼出来的——项目膨胀到看不清了，我不得不找一种方法让自己重新看见全局。试了各种办法，最后发现最管用的就是这个朴素的思路：给代码分好类，然后拍张照片。

如果你是一个有经验的程序员，你可能会觉得这本书讲的东西很基础。分层架构？模块划分？依赖管理？这些不是软件工程的常识吗？

是的。对专业开发者来说，这些确实是常识。

但对 Vibe Coder 来说，这些"常识"是全新的知识。我们不是从计算机课堂上走过来的，我们是从"AI 帮我实现一个想法"这个起点出发的。我们跳过了传统开发者花几年时间积累的那些基础认知，直接进入了产品开发。

这就是为什么 Vibe Coding 到一定阶段会撞墙——不是因为我们笨，而是因为我们缺少一些基础工具。Layer-Zone Tree 就是这样一个基础工具。它把"怎么组织代码"这件事，从专业开发者的隐性知识变成了一个 Vibe Coder 也能理解和使用的显性方法。

我不敢说这个方法是最好的。但我可以说它对我有用——六万多行代码的项目，我依然看得清全局，依然能精准地指挥 AI 干活。

## 关于 CLVZ

这本书专注讲 Layer-Zone Tree——怎么给代码分类、怎么拍全景照。但 Layer-Zone Tree 其实是一个更大框架的一部分。

这个框架叫 **CLVZ**——Contract-first, Layered, Validated, Zoned。四个字母，四个维度：

**C — Contract-first（契约优先）**

在动手写代码之前，先定义好 Zone 之间的协作接口。A Zone 需要 B Zone 提供什么？数据格式是什么？调用方式是什么？先把"契约"写清楚，再让 AI 按契约实现。这解决的是"怎么让 Zone 之间的协作不出错"的问题。

**L — Layered（分层）**

把代码按技术职责分层——界面、逻辑、数据。这本书第三章讲的内容。

**V — Validated（验证）**

怎么确保代码确实按照分层和分区的规则在运行？怎么检测 Zone 边界有没有被突破？怎么在 AI 写完代码之后自动验证合规性？这解决的是"怎么确保分类正确"的问题。

**Z — Zoned（分区）**

在每层内部按业务职责划分 Zone。这本书第四章和第五章讲的内容。

四个维度里，本书的重点是 **L 和 Z**——分层和分区。而 Layer-Zone Tree 是让你**看见** L 和 Z 的工具。它把分层和分区的结果可视化成一张全景照片，让你一眼看清项目长什么样。

C（契约）和 V（验证）是另外两个维度，它们解决的问题跟 Tree 不同。契约解决的是"Zone 之间怎么协作"，验证解决的是"怎么确保规则被遵守"。这两个话题都很重要，但它们不在本书的范围内——以后有机会专门聊。

**如果你只学一样东西——学 Layer-Zone Tree 就够了。**

因为所有问题的起点都是"看不见"。你看不见项目的结构，才会乱指挥。你看不见依赖关系，才会改 A 坏 B。你看不见哪个 Zone 在膨胀，才会等到它爆炸了才发现。

Tree 解决的是最根本的问题：让你看见。

看见了，你自然知道该怎么办。看见了 🔴 违规，你知道要解耦。看见了膨胀的 Zone，你知道要拆分。看见了迷路的文件，你知道要归位。

**看见，就是解决问题的一半。**

## GetDone Timer 还在进化

写这本书的时候，GetDone Timer 已经是一个六万多行代码的 macOS 应用了。任务管理、番茄钟计时、日程规划、AI 助手、统计分析、账户系统、支付系统——功能不少。

它还在继续长大。

但跟半年前不同的是，我不再害怕它长大了。

以前每次加一个大功能，我心里都有点发虚——"加完之后项目会不会更乱？我还能看得清吗？" 现在这种焦虑基本没有了。因为我知道，不管项目长到多大，我随时可以让 AI 拍一张全景照，几分钟之内看清全局。

新功能加完了？拍一张照。照片里一切正常？继续。某个 Zone 开始膨胀了？拆一下。出现了新的依赖违规？处理掉。

就这么简单。

Vibe Coding 的快乐没有减少一分。但焦虑少了很多。

## 给所有 Vibe Coder

如果你读到了这里，我猜你大概率是一个 Vibe Coder——或者至少是一个对 Vibe Coding 感兴趣的人。

不管你是刚开始用 AI 写代码的新手，还是已经 Vibe Coding 了好几个月、项目膨胀到开始发慌的"老手"，我想说的是同一句话：

**你不需要变成架构专家。**

你不需要读完 Clean Architecture，不需要搞懂 SOLID 原则的每一条，不需要知道六边形架构和洋葱架构的区别。这些知识有用，但不是必需的。

你需要的只是一张全景照。

给你的项目拍张照。看看它长什么样。有几层，每层有哪些 Zone，Zone 之间什么关系，有没有什么地方不对劲。

就这么多。

AI 会越来越强。它写代码的速度会越来越快，质量会越来越高，能处理的任务会越来越复杂。这意味着你的项目会以更快的速度膨胀。

但只要你手里有一张全景照——你就永远不会迷路。

**定期给你的项目拍张照吧。那张照片，就是你在 AI 时代最重要的导航仪。**
